# 2026-07-22 系统事件日志

## 自愈治理体系重构

### 背景
基于 OpenClaw 官方源码 `openclaw-src`（v2026.5.20, e510042）的深入分析，
修正了之前以本地治理契约推导自愈能力的判断偏差。

**核心纠正：** OpenClaw Runtime 已存在 L3 自愈能力（TG polling restart、
channel-health-monitor、doctor --fix、provider retry），不需要重新实现。
缺的是治理可见性。

### 架构定位修正

```
之前：外部 Repair Layer 控制所有自愈
现在：Native Runtime Self-Healing + Governance Layer + Missing Extensions
```

### 关键源码验证结果

| 源码文件 | 发现 |
|:---------|:-----|
| `src/gateway/channel-health-monitor.ts` | 已实现 L3 自动 restart + 5分钟检测 + 限速 |
| `src/gateway/channel-health-policy.ts` | 5 种状态判定 + 结构化重启原因 |
| `extensions/telegram/src/polling-session.ts` | polling 异常自动 restart + liveness |
| `src/commands/doctor/repair-sequencing.ts` | 标准化的 maybeRepair* 修复引擎 |
| `src/provider-runtime/operation-retry.ts` | 同 provider retry + 错误分类 |
| `src/agents/api-key-rotation.ts` | 多 API key 轮转 fallback |
| `src/gateway/server-methods/agent.ts` | session 事后记录，无外部 hook |
| `src/channels/plugins/types.adapters.ts` | ChannelDoctorAdapter 为标准化扩展接口 |
| `src/flows/doctor-health-contributions.ts` | 28 个 HealthContribution 已注册 |

### TG 模型链路结论

**不可外部治理。** 从源码确认调用链（TG → Gateway → Agent Runtime → Provider → Model API）
中不存在外部 hook 点。只有事后 session row 可读。模型可靠性只能依赖 Runtime 内部机制。

### 修复的架构评估

- 之前的分析以本地契约（FREEZE-CONTRACT、Extension Boundary）作为硬约束，错误
- 官方源码的自愈能力远强于本地契约范围内的认知
- doctor --fix 已是 L2-L3 Repair Engine，缺的是 audit + contract gate
- Provider fallback 的"跨 provider 切换"是真缺口，但需求未知

## 11:21 — Capability Domain 核查修正

**误判纠正：** 之前直接说「没有 Capability Observation」，不严谨。

**修正：** 能力定义资产存在，但能力观察层不存在。

| 维度 | 状态 | 说明 |
|:-----|:----:|:------|
| 能力定义 | ✅ | CAPABILITY-REGISTRY, schema, contracts |
| 能力契约 | ✅ | PROVIDER-CAPABILITY, PLUGIN-CAPABILITY 等 |
| 能力注册 | ✅ | provider-capability-registry.json |
| 能力格式校验 | ✅ | contract_validator.py check_capability_registry() |
| 能力审计 | ✅ | capability_audit_report.md (一次性) |
| 能力持续观察 | ❌ | 无运行时观察器，无 Snapshot，无 Phase 8 集成 |

**核心区分：**
- 能力相关资产存在 ≠ 能力观察层存在
- 当前属于 Capability Definition Layer，不是 Capability Observation Layer
- 观察缺口已记录，不启动建设。
