# 冻结契约 — Freeze Contract v1.0

> 定义冻结期的确切边界，防止错误理解导致审计缺口拖延。

## 冻结 ≠ 禁止修改

```
冻结 = Baseline State Protection
 |
 ├── 保护：已验证架构、已确认契约、已稳定行为
 ├── 允许：审计缺口修复、正确性修复、安全修复、数据完整性修复、边界覆盖补齐
 └── 禁止：无证据扩展、功能膨胀、重新设计
```

## 判定标准

每一项修改在进入冻结期后，必须回答：

```
1. 是否是审计发现的缺口？          → 允许修复
2. 是否是已有契约的边界未覆盖？      → 允许补齐
3. 是否是数据/安全/完整性问题？      → 允许修复
4. 是否新增了未规划的功能？          → 禁止
5. 是否涉及重新设计已有架构？        → 禁止
6. 是否无证据、仅凭推测的扩展？      → 禁止
```

**核心区分：修复 vs 扩展**

| 修复 | 扩展 |
|------|------|
| 关闭已有路径的绕过 | 增加新路径 |
| 补齐已有工具的能力缺口 | 创建新工具 |
| 恢复架构契约 | 订立新契约 |
| 覆盖已存在的消费场景 | 新增消费场景 |

## 反面案例（2026-07-20）

```
错误理解：
  冻结 → 不允许任何修改 → 审计缺口待冻结期结束后修复

正确：
  冻结 → 基线架构不扩展 → 审计缺口立即修复
```

## 验证方法

每次冻结期修改后，更新状态表：

```
Baseline:        🔒 Frozen
Audit Remediation:  ✅ 修复了什么
New Capability:     ❌ None
Integrity Fix:      ✅ Yes
```

状态表应附加在对应契约的末尾。
