# 冻结契约 — Freeze Contract v1.0

> ```yaml
> contract:
>   name: FREEZE-CONTRACT
>   version: v1.0
>   level: L0
>   parent: null
>   scope: system freeze rules
>   owner: 燃🔥
>   status: active
> ```
>
> 审计的目标不是发现异常，而是验证治理能力完整性。
> 任何缺口必须完成根因定位；若根因为治理定义缺失，则优先补齐治理定义，再执行修复。

## 冻结 ≠ 禁止修改

```
冻结 = Baseline State Protection
 |
 ├── 保护：已验证架构、已确认契约、已稳定行为
 ├── 允许：审计缺口修复、正确性修复、安全修复、数据完整性修复、边界覆盖补齐
 └── 禁止：无证据扩展、功能膨胀、重新设计
```

## 判定标准

每一项修改在进入冻结期后，必须回答：

```
1. 是否是审计发现的缺口？          → 允许修复
2. 是否是已有契约的边界未覆盖？      → 允许补齐
3. 是否是数据/安全/完整性问题？      → 允许修复
4. 是否新增了未规划的功能？          → 禁止
5. 是否涉及重新设计已有架构？        → 禁止
6. 是否无证据、仅凭推测的扩展？      → 禁止
```

**核心区分：修复 vs 扩展**

| 修复 | 扩展 |
|------|------|
| 关闭已有路径的绕过 | 增加新路径 |
| 补齐已有工具的能力缺口 | 创建新工具 |
| 恢复架构契约 | 订立新契约 |
| 覆盖已存在的消费场景 | 新增消费场景 |

## 审计发现三种类型

治理审计不处理 Bug。治理审计识别以下三类问题：

| 类型 | 含义 | 动作 |
|:----:|------|------|
| 🏗 **Governance Gap** | 规则/边界/契约缺失 | 补齐治理定义 → 验证 → 修复 |
| ⚖️ **Governance Violation** | 已有规则但运行偏离 | 定位责任 → 恢复一致性 |
| 🔍 **Valid Observation** | 行为符合设计，需长期观测 | 记录 → 持续观察 |

## 审计判断顺序（Evidence First Audit Procedure）

**未知不是观察理由。未知本身就是需要调查的状态。**

审计不靠经验推断设计意图。必须根据已批准的设计定义判断系统是否符合。
没有设计定义，就先解决设计定义缺失。

```
1. 收集事实                         ← 运行时证据，非模型推断
2. 查阅权威设计来源                 ← 契约 / 设计文档 / schema / 源码
3. 比对实际状态与设计定义           ← 差异即发现
4. 判断类型：
   ┌───────────────────────┐
   │ 设计是否存在？         │
   ├───────────┬───────────┤
   │ 有         │ 无        │
   ↓           ↓           │
   验证一致性   治理缺口     │
   ├───────────┴───────────┤
   │ 一致 → Valid Obs      │
   │ 偏离 → Violation      │
   └───────────────────────┘
5. 制定治理动作
6. 执行修复或补齐规则
7. 验证闭环
```

**禁止跳过第 2 步。** 设计未知不能作为观察理由，必须先查设计来源。

## 审计最高原则：根因追溯 + 治理闭环

**任何审计缺口都必须追溯根因，并完成治理闭环。缺口不能仅被记录、分类或观察，必须找到导致缺口产生的设计、契约、流程或实现原因，并修复根因。**

```
发现缺口
 ↓
根因分析（Root Cause Analysis）
 ↓
确认责任层
   ├── Governance Layer     → 补契约/规则
   ├── Data Contract Layer  → 补数据约束
   ├── Implementation Layer → 修复代码
   └── Process Layer        → 修复流程
 ↓
补齐治理规则或修复实现
 ↓
验证有效性
 ↓
关闭缺口
```

禁止模式：

```
发现：没有规则
结论：无法判断
处理：继续观察
```

**
缺少修复规则本身也是治理缺口（Governance Gap），应优先补齐规则，再执行修复。

统一术语：

| 不要说 | 改为 |
|--------|------|
| 审计发现问题 | 审计识别治理缺口或治理偏差 |
| 修复问题 | 补齐治理能力并恢复一致性 |
| Bug | 根据实际情况：Governance Gap / Violation / Valid Obs |

```
识别异常
 ↓
判断：设计是否存在？
 ↓
┌───────────┬───────────┐
│ 有 → 验证  │ 无 → Gap   │
│ 一致性      │ 补定义     │
│ ↓           │ ↓          │
│ 修复/Viol   │ 补规则+修复 │
└───────────┴───────────┘
 ↓
再次审计验证闭环
```

## 状态分层（按发现类型映射）

### 状态等级

| 状态 | 含义 | 典型对应发现类型 |
|:----:|------|:----------------:|
| 🟢 GREEN | 正常 | Valid Observation |
| 🟡 YELLOW | 需关注 | Governance Gap / Violation / Valid Obs |
| 🔴 RED | 需要处理 | Governance Violation（严重） |
| ⚪ UNKNOWN | 升级确认 | 无法通过已有来源判定 |

### YELLOW 内部分层

| 分层 | 发现类型 | 含义 | 动作 |
|:----:|:--------:|------|------|
| YELLOW-A | 🏗 Governance Gap | 规则/边界/契约缺失 | 补齐治理定义 → 执行修复 → 验证 |
| YELLOW-B | ⚖️ Governance Violation | 有规则未遵循 | 定位责任 → 恢复一致性 |
| YELLOW-C | 🔍 Valid Observation | 行为符合设计，需长期观测 | 记录 → 持续观察 |

**YELLOW-C 仅适用于：** 已查阅设计来源，设计确实允许当前行为。

**YELLOW-C 不适用于：** 设计未定义、设计来源不可达、未查阅设计来源。

**YELLOW-C 转 A/B 的条件：** 观察积累足够证据确认属于规则缺失或执行偏差时，应升级处理，不永久搁置。

## 验证方法

每次冻结期修改后，更新状态表：

```
Baseline:        🔒 Frozen
Audit Remediation:  ✅ 修复了什么
New Capability:     ❌ None
Integrity Fix:      ✅ Yes
```

状态表应附加在对应契约的末尾。
