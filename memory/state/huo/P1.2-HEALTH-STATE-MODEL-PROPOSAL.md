## 问题

P1.1 健康图使用二元状态模型：

```text
healthy / failed
```

判定依据是 mtime 与阈值比较。

但 event_kernel 案例暴露了缺陷：

```text
events/*.jsonl mtime > 2h
        ↓
mtime old
        ↓
当前判定：FAILED
实际语义：IDLE（无业务事件产生，但系统运行正常）
```

## 影响范围

**当前被影响节点：** event_kernel（仅此节点使用 mtime 判定）
**未来扩展节点：** 所有基于"新鲜度"判定的节点

## 提议

将健康状态模型从 2 级扩展为 5 级：

```text
ACTIVE     ─ 正常运行，最近有活动
IDLE       ─ 正常运行，但当前无业务事件（正常空闲）
STALE      ─ 超过预期阈值无活动，需关注
FAILED     ─ 确认异常
RECOVERING ─ 刚从 FAILED 恢复，正在验证
```

## 判定规则示例（event_kernel）

```text
events 目录 1h 内有写入     → ACTIVE
events 目录 1-24h 无写入    → IDLE（取决于节点类型）
events 目录 >24h 无写入     → STALE
events 目录无文件或写入错误   → FAILED
FAILED 后首次恢复写入        → RECOVERING（待下一周期确认）
```
